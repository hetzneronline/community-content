## Introduction

Garage is a lightweight, S3-compatible object storage server written in Rust by the Deuxfleurs association. It is designed to be distributed across several machines, but it runs perfectly well on a single server, which makes it a simple way to get an S3 endpoint of your own — for application uploads, backups (restic, Duplicati), static-site hosting, or anything that already speaks the S3 protocol.

This tutorial sets up a single Garage node with Docker, initializes its storage layout, and then creates a bucket and an access key. At the end you will upload, list, and download a file with the AWS CLI to confirm the endpoint behaves like S3.

Garage is free and open source (AGPL-3.0). This was tested on a fresh Ubuntu 26.04 LTS server with Garage v2.3.0.

**Prerequisites**

* One server (a fresh Ubuntu 24.04 or 26.04 install is fine)
* Root access, or a user with `sudo`

This tutorial uses the following example values:

* Username: `holu`
* IPv4 address: `10.0.0.1`
* Domain: `example.com`

Replace them with your own values. All commands below are run as `root`.

## Step 1 - Install Docker

We run Garage as a container. Install Docker and the Compose plugin with the official script, which works on both Ubuntu 24.04 and 26.04:

```bash
curl -fsSL https://get.docker.com -o get-docker.sh
sh get-docker.sh
```

Confirm it is there:

```bash
docker --version
```

## Step 2 - Write the Garage configuration

Garage reads a single TOML file. Create a working directory and the storage folders:

```bash
mkdir -p /opt/garage/meta /opt/garage/data
cd /opt/garage
```

Two values in the config must be secret. Generate a random RPC secret (nodes use it to talk to each other) and an admin token:

```bash
openssl rand -hex 32      # use this for rpc_secret
openssl rand -base64 32   # use this for admin_token
```

Create `/opt/garage/garage.toml` and paste the two values in:

```toml
metadata_dir = "/var/lib/garage/meta"
data_dir = "/var/lib/garage/data"
db_engine = "lmdb"

replication_factor = 1

rpc_bind_addr = "[::]:3901"
rpc_public_addr = "127.0.0.1:3901"
rpc_secret = "paste-the-openssl-hex-secret-here"

[s3_api]
s3_region = "garage"
api_bind_addr = "[::]:3900"
root_domain = ".s3.example.com"

[s3_web]
bind_addr = "[::]:3902"
root_domain = ".web.example.com"
index = "index.html"

[admin]
api_bind_addr = "[::]:3903"
admin_token = "paste-the-openssl-base64-token-here"
```

A few notes on these values:

- `replication_factor = 1` is what makes this a valid single-node setup. The default of 3 expects three servers and Garage refuses to start otherwise.
- Port `3900` is the S3 API, `3901` is internal RPC, `3902` serves buckets as websites, and `3903` is the admin API.
- `root_domain` under `[s3_api]` is only needed for virtual-host-style bucket URLs. We use path-style access in this tutorial, so you can leave the example domain as is for now.

## Step 3 - Start Garage

Create `/opt/garage/docker-compose.yml`. Using `network_mode: host` keeps the port setup simple since Garage listens on several ports:

```yaml
name: garage
services:
  garage:
    image: dxflrs/garage:v2.3.0
    container_name: garage
    restart: unless-stopped
    network_mode: host
    volumes:
      - ./garage.toml:/etc/garage.toml:ro
      - ./meta:/var/lib/garage/meta
      - ./data:/var/lib/garage/data
```

Start it:

```bash
docker compose up -d
```

## Step 4 - Initialize the cluster layout

A fresh node has storage but no assigned role, so it can't hold data yet. Check the status:

```bash
docker exec garage /garage status
```

The node shows up under `HEALTHY NODES` with `NO ROLE ASSIGNED`:

```
ID                Hostname        Address         Tags  Zone  Capacity          Version
3d2731dfff5459f9  holu            127.0.0.1:3901              NO ROLE ASSIGNED  v2.3.0
```

Copy the node ID from that output and assign it a zone and a capacity. Here we give it 1 GB; set this to whatever you want to allocate:

```bash
docker exec garage /garage layout assign -z dc1 -c 1G <node-id>
```

The change is only staged. Commit it:

```bash
docker exec garage /garage layout apply --version 1
```

Run `docker exec garage /garage status` again — the node now reports its capacity and available disk, which means it is ready to store objects.

## Step 5 - Create a bucket and an access key

Create a bucket:

```bash
docker exec garage /garage bucket create test-bucket
```

Create an access key for your application:

```bash
docker exec garage /garage key create test-app-key
```

Give that key full rights on the bucket:

```bash
docker exec garage /garage bucket allow --read --write --owner test-bucket --key test-app-key
```

The key's secret is shown only once at creation. If you missed it, ask Garage to print it again:

```bash
docker exec garage /garage key info --show-secret test-app-key
```

```
Key ID:       GK99cc099921c28c220fb9eaee
Secret key:   b718d70afbf9bad0265c6bc6691eddd085edecd8416574005629c0c5352205c3
```

Keep both values; you need them in the next step.

## Step 6 - Test the endpoint with the AWS CLI

Install the AWS CLI:

```bash
apt install -y awscli
```

Export the credentials from the previous step. `AWS_EC2_METADATA_DISABLED` stops the CLI from trying to reach a cloud metadata service that isn't relevant here:

```bash
export AWS_ACCESS_KEY_ID="GK99cc099921c28c220fb9eaee"
export AWS_SECRET_ACCESS_KEY="b718d70afbf9bad0265c6bc6691eddd085edecd8416574005629c0c5352205c3"
export AWS_EC2_METADATA_DISABLED=true
```

Upload a file, list the bucket, and download it again. The `--endpoint-url` flag points the CLI at Garage instead of AWS:

```bash
echo "Hello from Garage" > hello.txt
aws --endpoint-url http://localhost:3900 --region garage s3 cp hello.txt s3://test-bucket/hello.txt
aws --endpoint-url http://localhost:3900 --region garage s3 ls s3://test-bucket/
aws --endpoint-url http://localhost:3900 --region garage s3 cp s3://test-bucket/hello.txt downloaded.txt
```

The listing shows your object and the download returns its contents:

```
2026-06-17 13:12:51         62 hello.txt
```

You can confirm it from Garage's side too:

```bash
docker exec garage /garage bucket info test-bucket
```

The output now reports `Objects: 1` and the stored size.

## Step 7 - Reach Garage from other machines

So far the endpoint is `localhost`. To use it from another server or application, two things matter:

- Open the S3 port in the firewall: `ufw allow 3900/tcp`. Leave the RPC port `3901` and the admin port `3903` closed to the internet.
- Garage speaks plain HTTP. For anything beyond a trusted private network, put a reverse proxy with a TLS certificate in front of port `3900` and use that HTTPS address as your endpoint.

Then point your client at `http://<your-host>:3900` (or the HTTPS proxy address) with the same access key and secret.

## Conclusion

You now run your own S3-compatible object store on a single server and have verified it with real uploads and downloads. The same access keys work with any S3 client or SDK, so you can point backup tools, application file uploads, or a static website at it. When you outgrow one node, Garage is built to scale: add more servers, give each a role in the layout, and raise the replication factor.
