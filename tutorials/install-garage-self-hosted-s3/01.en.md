## Introduction

Garage is a lightweight, S3-compatible object storage server written in Rust by the Deuxfleurs association. It is built for distributed setups, but it also works well on a single server. That makes it a practical option when you want your own S3 endpoint for backups, application uploads, or static-site assets.

In this tutorial, you run Garage in Docker on one Ubuntu server, initialize the storage layout, create a bucket, and generate an access key. You then verify the setup by uploading, listing, and downloading an object with the AWS CLI.

Garage is free and open source (AGPL-3.0). The steps were tested on Ubuntu 26.04 LTS with Garage v2.3.0.

**Prerequisites**

* One server (a fresh Ubuntu 24.04 or 26.04 install is fine)
* Root access, or a user with `sudo`

This tutorial uses these example values:

* Username: `holu`
* IPv4 address: `10.0.0.1`
* Domain: `example.com`

Replace them with your own values. Run all commands as `root`.

## Step 1 - Install Docker

You will run Garage as a container. Install Docker and the Compose plugin with the official script:

```bash
curl -fsSL https://get.docker.com -o get-docker.sh
sh get-docker.sh
```

Verify that Docker is available:

```bash
docker --version
```

## Step 2 - Write the Garage configuration

Garage uses one TOML configuration file. First, create a working directory and data folders:

```bash
mkdir -p /opt/garage/meta /opt/garage/data
cd /opt/garage
```

Two values in the config must stay secret: the RPC secret (used for node-to-node communication) and the admin token. Generate both values now:

```bash
openssl rand -hex 32      # use this for rpc_secret
openssl rand -base64 32   # use this for admin_token
```

Create `/opt/garage/garage.toml` and paste your generated values:

```toml
metadata_dir = "/var/lib/garage/meta"
data_dir = "/var/lib/garage/data"
db_engine = "lmdb"

replication_factor = 1

rpc_bind_addr = "[::]:3901"
rpc_public_addr = "127.0.0.1:3901"
rpc_secret = "paste-the-openssl-hex-secret-here"

[s3_api]
s3_region = "garage"
api_bind_addr = "[::]:3900"
root_domain = ".s3.example.com"

[s3_web]
bind_addr = "[::]:3902"
root_domain = ".web.example.com"
index = "index.html"

[admin]
api_bind_addr = "[::]:3903"
admin_token = "paste-the-openssl-base64-token-here"
```

Why these settings matter:

- `replication_factor = 1` is required for a single-node setup. The default value `3` expects three nodes and Garage will not start with only one.
- Port `3900` is the S3 API, `3901` is internal RPC, `3902` serves buckets as websites, and `3903` is the admin API.
- `root_domain` under `[s3_api]` is only needed for virtual-host-style bucket URLs. In this tutorial, you use path-style access, so you can keep the example value for now.

## Step 3 - Start Garage

Create `/opt/garage/docker-compose.yml`. `network_mode: host` keeps networking simple because Garage listens on multiple ports:

```yaml
name: garage
services:
  garage:
    image: dxflrs/garage:v2.3.0
    container_name: garage
    restart: unless-stopped
    network_mode: host
    volumes:
      - ./garage.toml:/etc/garage.toml:ro
      - ./meta:/var/lib/garage/meta
      - ./data:/var/lib/garage/data
```

Start the container:

```bash
docker compose up -d
```

## Step 4 - Initialize the cluster layout

A fresh Garage node has no assigned role yet, so it cannot store data. Check the current status:

```bash
docker exec garage /garage status
```

You should see the node under `HEALTHY NODES` with `NO ROLE ASSIGNED`:

```
ID                Hostname        Address         Tags  Zone  Capacity          Version
3d2731dfff5459f9  holu            127.0.0.1:3901              NO ROLE ASSIGNED  v2.3.0
```

Copy the node ID from the output, then assign a zone and capacity. This example allocates 1 GB (adjust to your needs):

```bash
docker exec garage /garage layout assign -z dc1 -c 1G <node-id>
```

This change is only staged. Apply the layout:

```bash
docker exec garage /garage layout apply --version 1
```

Run `docker exec garage /garage status` again. The node should now show capacity and available disk, which means it is ready to store objects.

## Step 5 - Create a bucket and an access key

Create a bucket:

```bash
docker exec garage /garage bucket create test-bucket
```

Create an access key for your application:

```bash
docker exec garage /garage key create test-app-key
```

Grant full bucket permissions to that key:

```bash
docker exec garage /garage bucket allow --read --write --owner test-bucket --key test-app-key
```

Garage shows the secret once when the key is created. If you missed it, print it again:

```bash
docker exec garage /garage key info --show-secret test-app-key
```

```
Key ID:       GK99cc099921c28c220fb9eaee
Secret key:   b718d70afbf9bad0265c6bc6691eddd085edecd8416574005629c0c5352205c3
```

Save both values. You need them in the next step.

## Step 6 - Test the endpoint with the AWS CLI

Install the AWS CLI:

```bash
apt install -y awscli
```

Export the key and secret from the previous step. Set `AWS_EC2_METADATA_DISABLED=true` so the CLI does not try to query a cloud metadata service:

```bash
export AWS_ACCESS_KEY_ID="GK99cc099921c28c220fb9eaee"
export AWS_SECRET_ACCESS_KEY="b718d70afbf9bad0265c6bc6691eddd085edecd8416574005629c0c5352205c3"
export AWS_EC2_METADATA_DISABLED=true
```

Upload a file, list bucket contents, and download the file again. Use `--endpoint-url` to point the AWS CLI to Garage instead of AWS:

```bash
echo "Hello from Garage" > hello.txt
aws --endpoint-url http://localhost:3900 --region garage s3 cp hello.txt s3://test-bucket/hello.txt
aws --endpoint-url http://localhost:3900 --region garage s3 ls s3://test-bucket/
aws --endpoint-url http://localhost:3900 --region garage s3 cp s3://test-bucket/hello.txt downloaded.txt
```

The listing should show your object:

```
2026-06-17 13:12:51         62 hello.txt
```

You can also confirm the result from Garage:

```bash
docker exec garage /garage bucket info test-bucket
```

The output should now report `Objects: 1` and the stored size.

## Step 7 - Reach Garage from other machines

So far, you have used `localhost`. To access Garage from another machine or application, do the following:

- Open the S3 API port in your firewall: `ufw allow 3900/tcp`.
- Keep RPC (`3901`) and admin (`3903`) closed to the public internet.
- Garage serves plain HTTP by default. For public or untrusted networks, place a reverse proxy with TLS in front of port `3900` and use the HTTPS endpoint.

Then point your S3 client to `http://<your-host>:3900` (or your HTTPS proxy endpoint) using the same key and secret.

## Conclusion

You now have a working, self-hosted S3-compatible object store on a single server. You verified the setup with real uploads and downloads, so you can start using it with S3-compatible tools and SDKs right away.

When you need higher durability or capacity, add more nodes, assign them in the Garage layout, and increase the replication factor.
