## Introduction

This tutorial will guide you through the setup of a [Kubernetes](https://kubernetes.io) cluster on Hetzner Cloud servers. The resulting cluster will support the full range of Kubernetes objects including [LoadBalancer service types](https://kubernetes.io/docs/concepts/services-networking/service/#loadbalancer), [Persistent Volumes](https://kubernetes.io/docs/concepts/storage/persistent-volumes/) and a private network between the servers. It will not cover high availability of the Kubernetes control plane.

**Prerequisites**

* A [Hetzner Cloud](https://www.hetzner.com/cloud) account
* Familiarity with the [concepts of Kubernetes](https://kubernetes.io/docs/concepts/)
* Familiarity with Linux and working on the shell
* A local machine that has the following installed:
  * `ssh`
  * [`hcloud`](https://github.com/hetznercloud/cli/blob/main/docs/tutorials/setup-hcloud-cli.md)
  * [`kubectl`](https://kubernetes.io/docs/tasks/tools/install-kubectl-linux/)

*This tutorial was tested on Ubuntu 26.04 Hetzner Cloud servers and Kubernetes version v1.36.2*

## Step 1 - Create new Hetzner Cloud resources

For this tutorial the following resources will be used

* 1 Hetzner cloud network
* 1 Hetzner cloud server CX23
* 2 Hetzner cloud servers CX33
* 1 Hetzner Load Balancer

The CX23 server will be the master node and the CX33 servers the worker nodes.

> While this guide will create the resources manually, a production setup should be set up with terraform, ansible or comparable tools. Have a look at the tutorial "[Setup your own scalable Kubernetes cluster with the Terraform provider for Hetzner Cloud](https://community.hetzner.com/tutorials/setup-your-own-scalable-kubernetes-cluster)" for an example and guideline how to do so.

Create the required resources in the web interface or with the hcloud CLI tool

```bash
local$ hcloud network create --name kubernetes --ip-range 10.98.0.0/16
local$ hcloud network add-subnet kubernetes --network-zone eu-central --type server --ip-range 10.98.0.0/16
local$ hcloud server create --type cx23 --name master-1 --image ubuntu-26.04 --ssh-key <ssh_key_id> --network <network_id>
local$ hcloud server create --type cx33 --name worker-1 --image ubuntu-26.04 --ssh-key <ssh_key_id> --network <network_id>
local$ hcloud server create --type cx33 --name worker-2 --image ubuntu-26.04 --ssh-key <ssh_key_id> --network <network_id>
```

> The existing SSH keys can be listed with `hcloud ssh-key list`. The network ID is printed when creating the network and can be looked up with `hcloud network list`.

The names of the server do not affect the cluster creation, but should not be changed later on. Feel free to change the image and type regarding your needs.

### Step 1.1 - Update the servers (Optional)

It is recommended to update the servers after creation. Log onto each server and run

```bash
all$ apt-get update
all$ apt-get dist-upgrade
all$ reboot
```

## Step 2 - Configure the network

Each server will have a private and a public IP address configured. Those are noted as `<10.0.0.X>` as private and `<116.203.0.X>` as public IP address.

## Step 3 - Install Kubernetes

To install the Kubernetes cluster on the servers we will utilise [kubeadm](https://kubernetes.io/docs/setup/independent/create-cluster-kubeadm/). The interface between Kubernetes and the Hetzner Cloud will be the [Hetzner Cloud Controller Manager](https://github.com/hetznercloud/hcloud-cloud-controller-manager) and the [Hetzner Cloud Container Storage Interface](https://github.com/hetznercloud/csi-driver). Both tools are provided by the Hetzner Cloud team.

### Step 3.1 - Prepare the Cloud Controller Manager

The [Hetzner Cloud Controller Manager](https://github.com/hetznercloud/hcloud-cloud-controller-manager) requires the Kubernetes cluster to be set up to use an external cloud provider. This means `kubelet` has to be started with the `--cloud-provider=external` flag, so that the Cloud Controller Manager (and not kubelet itself) is responsible for assigning the node addresses and the `providerID`.

> The Kubernetes `.deb` packages ship a `/etc/default/kubelet` file which is referenced by the kubeadm systemd drop-in (`/usr/lib/systemd/system/kubelet.service.d/10-kubeadm.conf`) via `EnvironmentFile`. This file takes precedence over a custom `Environment=` drop-in, so we configure the extra kubelet arguments directly in `/etc/default/kubelet` once the packages are installed (see the **Configure kubelet** part of Step 3.2).

### Step 3.2 - Install containerd and Kubernetes Packages

As containerd and Kubernetes are installed on a distribution with systemd as init system (run `ps -p 1 -o comm=` to verify), containerd should be set up to use the systemd cgroups. To do so, download the `containerd.service` unit file on each server

```bash
all$ wget https://raw.githubusercontent.com/containerd/containerd/main/containerd.service
```
```bash
all$ mv containerd.service /usr/lib/systemd/system/
```

Then, reload the systemd unit files on all servers

```bash
all$ systemctl daemon-reload
```

Then, install the required packages by executing the following commands on all servers

> To double-check the architecture of your server, use:
> ```bash
> dpkg --print-architecture
> ```

- **Install [containerd](https://github.com/containerd/containerd/releases)**
  
  > Make sure to download the archive that matches your own architecture (`containerd-<VERSION>-<OS>-<ARCH>.tar.gz`)
  
  ```bash
  all$ wget https://github.com/containerd/containerd/releases/download/v2.3.1/containerd-2.3.1-linux-amd64.tar.gz
  ```
  ```bash
  all$ tar Czxvf /usr/local containerd-2.3.1-linux-amd64.tar.gz
  ```
  ```bash
  all$ systemctl enable --now containerd
  all$ systemctl status containerd
  ```

<br>

- **Install [runc](https://github.com/opencontainers/runc/releases)**
  
  > Make sure to download the binary that matches your own architecture (`runc.<ARCH>`)

  ```bash
  all$ wget https://github.com/opencontainers/runc/releases/download/v1.4.3/runc.amd64
  ```
  ```bash
  all$ install -m 755 runc.amd64 /usr/local/sbin/runc
  ```

<br>

- **Install [CNI plugins](https://github.com/containernetworking/plugins/releases)**
  
  > Make sure to download the archive that matches your own architecture (`cni-plugins-<OS>-<ARCH>-<VERSION>.tgz`)

  ```bash
  all$ wget https://github.com/containernetworking/plugins/releases/download/v1.9.1/cni-plugins-linux-amd64-v1.9.1.tgz
  ```
  ```bash
  all$ mkdir -p /opt/cni/bin
  all$ tar Czxvf /opt/cni/bin cni-plugins-linux-amd64-v1.9.1.tgz
  ```

  **Set up `config.toml`**
  
  ```bash
  all$ mkdir -p /etc/containerd/
  all$ containerd config default | sudo tee /etc/containerd/config.toml
  ```
  To use the systemd cgroup driver with runc, edit the `/etc/containerd/config.toml` file and set `SystemdCgroup` to "true":
  ```config
  [plugins."io.containerd.grpc.v1.cri".containerd.runtimes.runc]
    ...
    [plugins."io.containerd.grpc.v1.cri".containerd.runtimes.runc.options]
      SystemdCgroup = true
  ```
  ```bash
  all$ systemctl restart containerd
  ```

<br>

- **Install [Kubernetes](https://kubernetes.io/docs/tasks/tools/install-kubectl-linux/#install-using-native-package-management)**
  
  ```bash
  all$ mkdir -p /etc/apt/keyrings
  all$ curl -fsSL https://pkgs.k8s.io/core:/stable:/v1.36/deb/Release.key | gpg --dearmor -o /etc/apt/keyrings/kubernetes-apt-keyring.gpg
  ```
  ```bash
  all$ cat <<EOF >/etc/apt/sources.list.d/kubernetes.list
  deb [signed-by=/etc/apt/keyrings/kubernetes-apt-keyring.gpg] https://pkgs.k8s.io/core:/stable:/v1.36/deb/ /
  EOF
  ```
  ```bash
  all$ apt-get update
  all$ apt-get install kubeadm kubectl kubelet
  ```

  > To install a different Kubernetes minor version, replace `v1.36` in both commands with the version you want (see the [available releases](https://kubernetes.io/releases/)). Always use the same minor version on every node.

<br>

- **Configure kubelet**

  Configure kubelet to use the external cloud provider and to register the node with its **private** IP address. The Kubernetes packages ship a `/etc/default/kubelet` file that is sourced by the kubeadm systemd unit, so set the arguments there. Replace `<private_ip>` with the actual private IP of *each* server (for example `10.98.0.2`):

  ```bash
  all$ cat <<EOF >/etc/default/kubelet
  KUBELET_EXTRA_ARGS=--cloud-provider=external --node-ip=<private_ip>
  EOF
  ```
  Then reload systemd and restart kubelet:
  ```bash
  all$ systemctl daemon-reload && systemctl restart kubelet
  ```

  > `--cloud-provider=external` hands node initialization (addresses, `providerID`) over to the Hetzner Cloud Controller Manager, while `--node-ip` makes the node's `InternalIP` - and therefore the flannel overlay traffic between nodes - use the private network. kubelet will not become fully ready until `kubeadm init`/`join` has run; until then a restart loop is expected.

<br>

- **Set the sysctl settings**

  You need to make sure that the system can actually forward traffic between the nodes and pods. Set the following sysctl settings on each server
  
  ```bash
  all$ cat <<EOF | tee /etc/modules-load.d/k8s.conf
  overlay
  br_netfilter
  EOF
  ```
  ```bash
  all$ modprobe overlay
  all$ modprobe br_netfilter
  ```
  ```bash
  all$ cat <<EOF | tee /etc/sysctl.d/k8s.conf
  # Allow IP forwarding for kubernetes
  net.bridge.bridge-nf-call-iptables = 1
  net.ipv4.ip_forward                = 1
  net.ipv6.conf.default.forwarding   = 1
  EOF
  ```
  ```bash
  all$ sysctl --system
  ```
  
  > These settings will allow forwarding of IPv4 and IPv6 packages between multiple network interfaces. This is required because each container has its own virtual network interface.

### Step 3.3 - Setup control plane

The servers are now prepared to finally install the Kubernetes cluster. Log on to the master node and initialize the cluster

> See [available Kubernetes versions](https://kubernetes.io/releases/)

```bash
master$ kubeadm config images pull
master$ kubeadm init \
  --pod-network-cidr=10.244.0.0/16 \
  --kubernetes-version=v1.36.2 \
  --upload-certs \
  --apiserver-cert-extra-sans <private_ip_master>
```

> Replace `<private_ip_master>` with the private IP of the master node (for example `10.98.0.2`) and `--kubernetes-version` with the minor version you installed in Step 3.2.

> If the first command does not work, execute `containerd config default > /etc/containerd/config.toml` to reset the containerd configuration and try again.

The `kubeadm init` process will print a `kubeadm join` command in between. You should copy that command for later use (not required as you can always create a new token when needed). The `--apiserver-cert-extra-sans` flag ensures your internal IP is recognized as valid IP for the apiserver.

> If the server is CX11, the `--ignore-preflight-errors=NumCPU` flag needs to be added. You can add additional flags regarding your needs. Check the [documentation](https://kubernetes.io/docs/reference/setup-tools/kubeadm/kubeadm-init/#options) for further information.

When the initialisation is complete, begin with setting up the required master components in the cluster. For ease of use, configure the kubeconfig of the root user to use the admin config of the Kubernetes cluster

```bash
master$ mkdir -p /root/.kube
master$ cp -i /etc/kubernetes/admin.conf /root/.kube/config
```

The cloud controller manager and the container storage interface require a secret in the `kube-system` namespace containing the access token for the Hetzner Cloud API

```bash
master$ cat <<EOF | kubectl apply -f -
apiVersion: v1
kind: Secret
metadata:
  name: hcloud
  namespace: kube-system
stringData:
  token: "<hetzner_api_token>"
  network: "<hetzner_network_id>"
EOF
```

> To create a Hetzner Cloud API token, see [the official getting started](https://docs.hetzner.com/cloud/api/getting-started/generating-api-token). You will **not** be able to fetch the secret key again later on, so don't close the popup before you have copied the token.

Now deploy the Hetzner Cloud Controller Manager into the cluster. Because this tutorial uses an overlay CNI (flannel, see below), deploy the plain `ccm.yaml` manifest - **not** `ccm-networks.yaml`:

```bash
master$ kubectl apply -f https://github.com/hetznercloud/hcloud-cloud-controller-manager/releases/download/v1.32.0/ccm.yaml
```

> `ccm-networks.yaml` additionally enables the Kubernetes route controller (`--allocate-node-cidrs`/native routing through the Hetzner network). That only works if the Pod CIDR is part of your Hetzner network range and is meant to be used *without* an overlay CNI. Combined with flannel it cannot create the routes and will mark your nodes as `NetworkUnavailable`, so we use `ccm.yaml` instead. See the [latest release](https://github.com/hetznercloud/hcloud-cloud-controller-manager/releases) for newer versions.

To let the Cloud Controller Manager attach `LoadBalancer` Services to your private network and report the private IP as the node's `InternalIP`, make it aware of the network by adding the `HCLOUD_NETWORK` environment variable (it reads the network from the `hcloud` Secret you created above):

```bash
master$ kubectl -n kube-system patch deployment hcloud-cloud-controller-manager --type=json \
  -p '[{"op":"add","path":"/spec/template/spec/containers/0/env/-","value":{"name":"HCLOUD_NETWORK","valueFrom":{"secretKeyRef":{"name":"hcloud","key":"network"}}}}]'
```

And set up the cluster networking

```bash
master$ kubectl apply -f https://github.com/flannel-io/flannel/releases/download/v0.28.5/kube-flannel.yml
```

> This tutorial uses flannel, as the CNI has very low maintenance requirements. The flannel project moved from `coreos/flannel` to [`flannel-io/flannel`](https://github.com/flannel-io/flannel); check the [releases page](https://github.com/flannel-io/flannel/releases) for the current version. flannel's default network (`10.244.0.0/16`) matches the `--pod-network-cidr` used above. For other options and comparisons check the [official documentation](https://kubernetes.io/docs/setup/independent/create-cluster-kubeadm/#pod-network)

As Kubernetes with the external cloud provider flag activated will add a taint to uninitialized nodes, the cluster critical pods need to be patched to tolerate these

```bash
master$ kubectl -n kube-flannel patch ds kube-flannel-ds --type json -p '[{"op":"add","path":"/spec/template/spec/tolerations/-","value":{"key":"node.cloudprovider.kubernetes.io/uninitialized","value":"true","effect":"NoSchedule"}}]'
master$ kubectl -n kube-system patch deployment coredns --type json -p '[{"op":"add","path":"/spec/template/spec/tolerations/-","value":{"key":"node.cloudprovider.kubernetes.io/uninitialized","value":"true","effect":"NoSchedule"}}]'
```

> Taints are flags on a node, that only specific sets of pods are allowed to be scheduled on that node (which is defined by the pod tolerations). The controller-manager will add a `node.cloudprovider.kubernetes.io/uninitialized` taint to each node which is not yet initialized by the controller-manager. But for the initialization to succeed the networking on the node and Cluster-DNS must be functional. Further information on taints and tolerations can be found in the [documentation](https://kubernetes.io/docs/concepts/configuration/taint-and-toleration/).

> If you just use a single node cluster, take care to taint the master node to accept pods: `kubectl taint nodes --all node-role.kubernetes.io/master-`

Last but not least deploy the Hetzner Cloud Container Storage Interface to the cluster

```bash
master$ kubectl apply -f https://raw.githubusercontent.com/hetznercloud/csi-driver/v2.21.2/deploy/kubernetes/hcloud-csi.yml
```

> See the [latest CSI driver release](https://github.com/hetznercloud/csi-driver/releases) for newer versions. This sets `hcloud-volumes` as the default `StorageClass`, so a `PersistentVolumeClaim` will automatically provision and attach a Hetzner Cloud Volume.

Your control plane is now ready to use. Fetch the kubeconfig from the master server to be able to use `kubectl` locally

```bash
local$ scp root@<116.203.0.1>:/etc/kubernetes/admin.conf ${HOME}/.kube/config
```

Or merge your existing kubeconfig with the `admin.conf` accordingly.

### Step 3.4 - Secure nodes

Using the Hetzner Firewall you can secure your nodes. Replace `<116.203.0.x>` with the public IPs of your node servers.

```bash
local$ hcloud firewall add-rule k8s-nodes --protocol=tcp --direction=in --source-ips <116.203.0.1>/32 --source-ips <116.203.0.2>/32 --source-ips <116.203.0.3>/32 --port any
local$ hcloud firewall add-rule k8s-nodes --protocol=udp --direction=in --source-ips <116.203.0.1>/32 --source-ips <116.203.0.2>/32 --source-ips <116.203.0.3>/32 --port any
```

### Step 3.5 - Join worker nodes

In the `kubeadm init` process a join command for the worker nodes was printed. If you don't have that command noted anymore, a new one can be generated by running the following command on the master node

```bash
master$ kubeadm token create --print-join-command
```

Then log on to each worker node and execute the join command

```bash
worker$ kubeadm join <116.203.0.1>:6443 --token <token> --discovery-token-ca-cert-hash sha256:<hash>
```

When the join was successful list all nodes

```bash
local$ kubectl get nodes -o wide
NAME       STATUS   ROLES           AGE   VERSION   INTERNAL-IP   EXTERNAL-IP       ...
master-1   Ready    control-plane   11m   v1.36.2   10.98.0.2     167.233.129.81    ...
worker-1   Ready    <none>          5m    v1.36.2   10.98.0.3     167.233.135.142   ...
worker-2   Ready    <none>          5m    v1.36.2   10.98.0.4     167.233.138.77    ...
```

> Note how the `INTERNAL-IP` column shows the **private** addresses (`10.98.0.x`) - this confirms the `--node-ip` flag and the Cloud Controller Manager are working together as intended.

### Step 3.6 - Setup LoadBalancing (Optional)

Hetzner Cloud supports [LoadBalancer as a Service](https://kubernetes.io/docs/concepts/services-networking/service/#loadbalancer). For more information about the option to use Hetzner Cloud Load Balancers with Kubernetes Services, see [Kubernetes Cloud Controller Manager for Hetzner Cloud](https://github.com/hetznercloud/hcloud-cloud-controller-manager?tab=readme-ov-file#annotation-and-environment-variable-reference-generation).

First, check if your nodes have the correct provider ID:

```shell
local$ kubectl describe node master-1 | grep "ProviderID"
local$ kubectl describe node worker-1 | grep "ProviderID"
local$ kubectl describe node worker-2 | grep "ProviderID"
```

The output should be `hcloud://<server_id>`.

<blockquote>
<details>
<summary>If the provider ID is missing, you can add it manually as a quick fix.</summary>

```bash
local$ hcloud server list | grep kubernetes
```
You should get output like this:
```bash
<server_id>   <name>   <status>   <ipv4>   <ipv6>   <private_net>   <location>   <age>
```
In the commands below, replace `<server_id>` with the actual server ID.
```bash
local$ kubectl patch node master-1 -p '{"spec":{"providerID":"hcloud://<server_id>"}}'
local$ kubectl patch node worker-1 -p '{"spec":{"providerID":"hcloud://<server_id>"}}'
local$ kubectl patch node worker-2 -p '{"spec":{"providerID":"hcloud://<server_id>"}}'
```

</details>
</blockquote>

If they have the provider ID, you can test the Load Balancer with this example deployment:

```bash
local$ nano example-service.yaml
```
```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: simple-web
spec:
  replicas: 3
  selector:
    matchLabels:
      app: ingress-nginx
  template:
    metadata:
      labels:
        app: ingress-nginx
    spec:
      containers:
      - name: simple-web
        image: httpd:latest
        ports:
        - containerPort: 80
---
apiVersion: v1
kind: Service
metadata:
  name: ingress-nginx-controller
  annotations:
    load-balancer.hetzner.cloud/name: "kubernetes-load-balancer"
    load-balancer.hetzner.cloud/location: "fsn1"
    load-balancer.hetzner.cloud/use-private-ip: "true"
    # If you have a public domain and a SSL certificate, upload the
    # SSL certificate in Hetzner Console and uncomment the lines below.
    #load-balancer.hetzner.cloud/protocol: "https"
    #load-balancer.hetzner.cloud/http-redirect-http: "true"
    #load-balancer.hetzner.cloud/http-certificates: "certificate-name"
spec:
  type: LoadBalancer
  selector:
    app: ingress-nginx
  ports:
    - protocol: TCP
      # If you have a public domain and a SSL certificate,
      # use port 443 instead of port 80.
      port: 80
      targetPort: 80
```

Next, apply the file. Note that this will create a new Load Balancer with the name `kubernetes-load-balancer`.

```bash
local$ kubectl apply -f example-service.yaml
local$ kubectl get pods
local$ kubectl get deployments
local$ kubectl get svc
```

If everything worked as expected, you can go to `http://<load_balancer_ip>` and it should say "It works!"

When you're done, you can remove the deployment and Load Balancer:

```bash
local$ kubectl delete -f example-service.yaml
```

### Step 3.7 - Cheaper LoadBalancing with a Floating IP (Optional)

A Hetzner Cloud Load Balancer is the most robust option, but it is also the most expensive part of this setup. If you only need a single, stable public entrypoint (for example an ingress controller) and can accept a short failover delay, a [Floating IP](https://docs.hetzner.com/cloud/floating-ips/overview) combined with the [`hcloud-fip-controller`](https://github.com/cbeneke/hcloud-fip-controller) is a much cheaper alternative.

How it compares to the Load Balancer:

| | Hetzner Load Balancer | Floating IP + `hcloud-fip-controller` |
|---|---|---|
| Approx. price | from ~5 €/month (LB11) | ~1 €/month (one IPv4 Floating IP) |
| Traffic distribution | actively balances across **all** healthy nodes | only **one** node serves the IP at a time |
| Health checks | built-in, per-target | none - relies on the node dropping out of the cluster |
| Failover | seconds | minutes by default (see caveats below) |
| TLS termination | offloaded to the LB | handled inside the cluster (e.g. by your ingress) |

The controller runs as a small Deployment that uses leader election. It periodically checks the Hetzner Cloud API and makes sure the configured Floating IP is assigned to a node that currently runs a controller pod. If that node leaves the cluster, the Floating IP is re-assigned to another node.

**Create a Floating IP**

```bash
local$ hcloud floating-ip create --type ipv4 --home-location fsn1 --name k8s-fip
```

> Note the assigned IP address - it is referenced as `<floating_ip>` below.

**Configure the Floating IP on every node**

The controller only manages the *Hetzner API assignment*; it does **not** configure the IP inside the operating system. So the Floating IP has to be present on the network interface of **every** node that could hold it, otherwise the node will not answer traffic after a failover. On Ubuntu (netplan), create `/etc/netplan/60-floating-ip.yaml` on each node:

```yaml
network:
  version: 2
  ethernets:
    eth0:
      addresses:
        - <floating_ip>/32
```
```bash
all$ chmod 600 /etc/netplan/60-floating-ip.yaml
all$ netplan apply
```

**Deploy the controller**

Create the namespace and the RBAC resources:

```bash
master$ kubectl create namespace fip-controller
master$ kubectl apply -f https://raw.githubusercontent.com/cbeneke/hcloud-fip-controller/master/deploy/rbac.yaml
```

Provide the configuration (the Floating IP) and a Hetzner Cloud API token. You can reuse the token from the `hcloud` Secret created earlier:

```bash
master$ cat <<EOF | kubectl apply -f -
apiVersion: v1
kind: ConfigMap
metadata:
  name: fip-controller-config
  namespace: fip-controller
data:
  config.json: |
    {
      "hcloud_floating_ips": [ "<floating_ip>" ]
    }
---
apiVersion: v1
kind: Secret
metadata:
  name: fip-controller-secrets
  namespace: fip-controller
stringData:
  HCLOUD_API_TOKEN: "<hetzner_api_token>"
EOF
```

Then deploy the controller itself. This is based on the upstream `deploy/deployment.yaml`, pinned to the latest release and with the ConfigMap mounted:

```bash
master$ cat <<EOF | kubectl apply -f -
apiVersion: apps/v1
kind: Deployment
metadata:
  name: fip-controller
  namespace: fip-controller
spec:
  replicas: 3
  selector:
    matchLabels:
      app: fip-controller
  template:
    metadata:
      labels:
        app: fip-controller
    spec:
      serviceAccountName: fip-controller
      containers:
        - name: fip-controller
          image: cbeneke/hcloud-fip-controller:v0.5.0
          env:
            - name: NODE_NAME
              valueFrom:
                fieldRef:
                  fieldPath: spec.nodeName
            - name: POD_NAME
              valueFrom:
                fieldRef:
                  fieldPath: metadata.name
            - name: NAMESPACE
              valueFrom:
                fieldRef:
                  fieldPath: metadata.namespace
          envFrom:
            - secretRef:
                name: fip-controller-secrets
          volumeMounts:
            - name: config
              mountPath: /app/config
      volumes:
        - name: config
          configMap:
            name: fip-controller-config
EOF
```

After a few seconds the controller assigns the Floating IP to one of the nodes. You can verify this with:

```bash
local$ hcloud floating-ip describe k8s-fip
master$ kubectl -n fip-controller logs -l app=fip-controller --tail=20
```

To actually serve traffic on the Floating IP, run your ingress controller (or any other entrypoint) so that it listens on every node, e.g. as a `DaemonSet` with `hostNetwork: true` or via a `NodePort` Service. Requests to `<floating_ip>` then reach whichever node currently holds the IP.

> **Failover caveats (the "less HA" part).** The controller considers a node a valid target as long as one of its pods is scheduled there. When a node fails *hard* (e.g. it is powered off but not deleted), Kubernetes keeps the pod on the unreachable node in `Terminating` state for `tolerationSeconds` (5 minutes by default) before it is removed - and only then does the Floating IP move. To get faster failover, lower the `node.kubernetes.io/unreachable` and `node.kubernetes.io/not-ready` toleration seconds on the controller pods, or run the controller as a `DaemonSet`. Either way, because only a single node holds the IP at any time, this is a cost-optimized failover solution, not a true highly-available, load-distributing setup like the Hetzner Load Balancer.

When you are done testing, you can remove everything again:

```bash
master$ kubectl delete namespace fip-controller
local$ hcloud floating-ip delete k8s-fip
```

## Conclusion

Congratulations, you now got a fully featured Kubernetes cluster running on Hetzner Cloud servers. Now you should start to install your first applications on the cluster or look into [high availability for your control plane](https://kubernetes.io/docs/setup/independent/high-availability/).
