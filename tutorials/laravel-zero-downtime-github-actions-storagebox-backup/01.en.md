## Introduction

Every time you run `git pull` on a live server, there is a window — sometimes a few seconds,
sometimes longer — where Nginx is serving a mix of old and new code. Old cached views with
new routes, new controller logic against a not-yet-migrated database column, Composer's
autoloader in the middle of being rebuilt. Your users hit that window and see errors.

This tutorial solves that problem without a paid tool like Laravel Forge or Envoyer. You
will set up a deployment system based on the **atomic symlink pattern** — the same approach
Capistrano made famous — where a new release is fully prepared in a staging directory before
a single `ln -sfn` command instantly switches production traffic to it. If anything goes wrong,
rolling back to the previous release is one command.

On top of that, you will configure a nightly cron job that dumps your MySQL database, encrypts
the dump with GPG, and ships it over SSH to a **Hetzner Storage Box** — offsite, encrypted
backup storage that is billed separately from your server and survives even if the server itself
is destroyed.

**What you will build:**

- An atomic release directory structure on your server (`releases/`, `current/`, `shared/`)
- A GitHub Actions workflow triggered on push to `main` that deploys a new release with zero
  downtime and signals queue workers to restart gracefully
- Environment-specific branches: `main` deploys to production, `staging` deploys to a staging
  path on the same server
- Instant one-command rollback that repoints the live symlink to the previous release
- A nightly cron job that dumps MySQL, GPG-encrypts the dump, and uploads it to a Hetzner
  Storage Box, keeping 7 days of backups and pruning older ones automatically

**What this tutorial assumes:**

- You already have a Laravel application running on Ubuntu 24.04 with Nginx and PHP-FPM.
  If not, set that up first before continuing.
- Your application code lives in a GitHub repository with a `main` branch for production and a
  `staging` branch for staging.
- You have a Hetzner Cloud server and a Hetzner Storage Box. Storage Boxes can be ordered
  from [hetzner.com/storage/storage-box](https://www.hetzner.com/storage/storage-box).

**Example terminology**

| Placeholder | Meaning |
|---|---|
| `<10.0.0.1>` | Your server's public IPv4 address |
| `<example.com>` | Your production domain |
| `<staging.example.com>` | Your staging domain |
| `holu` | The non-root Linux user that owns the application |
| `your-app` | Your application name, used as a directory name |
| `u123456` | Your Hetzner Storage Box username |
| `u123456.your-storagebox.de` | Your Hetzner Storage Box hostname |

## Step 1 - Understanding the Release Directory Structure

Before touching the server, it helps to understand what you are building and why it eliminates
downtime.

A standard `git pull` deployment looks like this on disk:

```
/var/www/your-app/         ← Nginx root points here
    .env
    storage/
    public/
    app/
    ...all application files...
```

During a deployment, files are overwritten one by one. Nginx keeps serving from this directory
the whole time, so it can serve a mix of old and new files mid-deployment.

The atomic pattern looks like this instead:

```
/var/www/your-app/
    releases/
        20250516143000/    ← a completed new release (not live yet)
        20250509091500/    ← previous release (still intact for rollback)
        20250501120000/    ← older release
    shared/
        .env               ← shared across all releases, never duplicated
        storage/           ← Laravel's storage directory, shared across all releases
    current -> releases/20250516143000/   ← symlink, points to the live release
```

Nginx's `root` points to `/var/www/your-app/current/public`. When a new release is ready,
`ln -sfn` replaces the `current` symlink atomically — the kernel swaps the pointer in one
operation, so there is no moment where `current` points to nothing or to a half-built release.

The `shared/` directory holds files that must persist across releases and be the same in every
release: your `.env` file and Laravel's `storage/` directory (logs, uploaded files, compiled
views). Each release gets a symlink to these shared resources rather than its own copy.

## Step 2 - Prepare the Server Directory Structure

SSH into your server as `holu`:

```bash
ssh holu@<10.0.0.1>
```

Create the directory layout:

```bash
sudo mkdir -p /var/www/your-app/releases
sudo mkdir -p /var/www/your-app/shared/storage/app/public
sudo mkdir -p /var/www/your-app/shared/storage/framework/{cache,sessions,views}
sudo mkdir -p /var/www/your-app/shared/storage/logs
sudo chown -R holu:holu /var/www/your-app
```

Copy your existing `.env` file into `shared/`. This is the single authoritative copy that every
release will use:

```bash
cp /var/www/your-app/.env /var/www/your-app/shared/.env
```

> If you do not have an existing `.env` yet, create one now with your production values. This
> file is never committed to Git. It stays on the server in `shared/` permanently.

Update Nginx to point to the `current/public` symlink instead of a fixed path. Open your
site config:

```bash
sudo nano /etc/nginx/sites-available/your-app
```

Change the `root` line:

```nginx
# Before
root /var/www/your-app/public;

# After
root /var/www/your-app/current/public;
```

Test and reload Nginx:

```bash
sudo nginx -t && sudo systemctl reload nginx
```

> Nginx will return a warning that `current/public` does not exist yet. That is expected — the
> symlink will be created by the first deployment in the next steps.

## Step 3 - Create the Deployment Script on the Server

The GitHub Actions runner will SSH into your server and execute a deployment script. Keeping
the logic in a script on the server — rather than inlining shell commands in the YAML workflow
— means you can test and edit the deployment without touching GitHub every time.

Create the script:

```bash
mkdir -p /var/www/your-app/scripts
nano /var/www/your-app/scripts/deploy.sh
```

Paste the following. Read each section — the inline comments explain every decision:

```bash
#!/usr/bin/env bash
set -euo pipefail

# ── Configuration ────────────────────────────────────────────────────────────
APP_DIR="/var/www/your-app"
REPO="git@github.com:your-username/your-laravel-app.git"
BRANCH="${1:-main}"          # Branch is passed as the first argument by GitHub Actions.
                             # Defaults to main if called manually.
KEEP_RELEASES=5              # How many old releases to keep. Older ones are pruned.
TIMESTAMP=$(date +"%Y%m%d%H%M%S")
RELEASE_DIR="$APP_DIR/releases/$TIMESTAMP"
SHARED_DIR="$APP_DIR/shared"
CURRENT_LINK="$APP_DIR/current"
# ─────────────────────────────────────────────────────────────────────────────

echo "==> Deploying branch '$BRANCH' as release $TIMESTAMP"

# 1. Clone the target branch into a fresh timestamped release directory.
#    --depth 1 skips the full Git history, which speeds up the clone.
git clone --depth 1 --branch "$BRANCH" "$REPO" "$RELEASE_DIR"

echo "==> Linking shared resources"

# 2. Remove the placeholder directories that Laravel creates in a fresh clone
#    so we can replace them with symlinks to the shared copies.
rm -rf "$RELEASE_DIR/storage"
rm -f  "$RELEASE_DIR/.env"

# 3. Symlink shared .env and storage into the new release.
#    The application sees them as if they were its own files.
ln -sfn "$SHARED_DIR/storage" "$RELEASE_DIR/storage"
ln -sfn "$SHARED_DIR/.env"    "$RELEASE_DIR/.env"

echo "==> Installing Composer dependencies"

# 4. Install PHP dependencies inside the new release directory.
#    --no-dev:                skips development packages (PHPUnit, etc.)
#    --optimize-autoloader:   generates a classmap for faster autoloading
#    --no-interaction:        suppresses any interactive prompts
composer install \
  --working-dir="$RELEASE_DIR" \
  --no-dev \
  --optimize-autoloader \
  --no-interaction \
  --quiet

echo "==> Setting permissions"

# 5. PHP-FPM runs as holu. The storage and bootstrap/cache directories
#    must be writable. Apply this to the shared storage (one location)
#    and to bootstrap/cache inside the new release.
chmod -R 775 "$SHARED_DIR/storage"
chmod -R 775 "$RELEASE_DIR/bootstrap/cache"

echo "==> Running database migrations"

# 6. Run migrations against the live database. The new release is not live
#    yet, but migrations run first so the database is ready when the symlink
#    flips. Write your migrations to be backward-compatible with the previous
#    release (add columns before removing old ones) to keep this safe.
php "$RELEASE_DIR/artisan" migrate --force

echo "==> Caching config, routes, and views"

# 7. Pre-compile the config, route list, and Blade templates.
#    These compiled files are stored in bootstrap/cache/ inside the release.
php "$RELEASE_DIR/artisan" config:cache
php "$RELEASE_DIR/artisan" route:cache
php "$RELEASE_DIR/artisan" view:cache

echo "==> Activating release (atomic symlink swap)"

# 8. This is the zero-downtime step. ln -sfn replaces the current symlink
#    atomically. From Nginx's perspective, the new release is live the instant
#    this command completes. There is no window where current is broken.
ln -sfn "$RELEASE_DIR" "$CURRENT_LINK"

echo "==> Restarting queue workers gracefully"

# 9. Signal running queue workers to finish their current job and then stop.
#    Supervisor will restart them, and they will pick up the new code.
#    This is safer than a hard restart, which can kill a job mid-execution.
php "$CURRENT_LINK/artisan" queue:restart

echo "==> Pruning old releases (keeping $KEEP_RELEASES)"

# 10. Delete releases older than the most recent KEEP_RELEASES.
#     The pipeline: list all release dirs sorted by name (which is a timestamp,
#     so alphabetical = chronological), reverse to get newest first, skip the
#     first KEEP_RELEASES, then delete the rest.
ls -1d "$APP_DIR/releases"/*/ \
  | sort -r \
  | tail -n +$((KEEP_RELEASES + 1)) \
  | xargs -r rm -rf

echo "==> Deployment complete: $RELEASE_DIR is now live"
```

Make the script executable:

```bash
chmod +x /var/www/your-app/scripts/deploy.sh
```

## Step 4 - Create a Dedicated Deploy SSH Key

GitHub Actions needs to SSH into your server to trigger the deployment script. You should
create a dedicated SSH key pair for this purpose — not your personal key — so you can revoke
it independently and it has a clear audit trail.

### Step 4.1 - Generate the Key Pair

Run this on your **local machine** (not the server):

```bash
ssh-keygen -t ed25519 -C "github-actions-deploy" -f ~/.ssh/github_actions_deploy
```

When prompted for a passphrase, press **Enter** for no passphrase. GitHub Actions cannot
interactively enter a passphrase, so the key must be unencrypted.

This creates two files:
- `~/.ssh/github_actions_deploy` — the private key (goes into GitHub)
- `~/.ssh/github_actions_deploy.pub` — the public key (goes onto the server)

### Step 4.2 - Add the Public Key to the Server

Copy the public key to your server's authorized keys:

```bash
ssh-copy-id -i ~/.ssh/github_actions_deploy.pub holu@<10.0.0.1>
```

Or manually append it:

```bash
cat ~/.ssh/github_actions_deploy.pub | ssh holu@<10.0.0.1> "cat >> ~/.ssh/authorized_keys"
```

### Step 4.3 - Restrict What the Key Can Do

Open `authorized_keys` on the server and restrict the deploy key so that even if it were
stolen, it can only run the deployment script and nothing else:

```bash
nano ~/.ssh/authorized_keys
```

Find the line that starts with `ssh-ed25519` and has the comment `github-actions-deploy`.
Prepend a `command=` restriction to it:

```
command="/var/www/your-app/scripts/deploy.sh ${SSH_ORIGINAL_COMMAND#* }",no-port-forwarding,no-X11-forwarding,no-agent-forwarding,no-pty ssh-ed25519 AAAA... github-actions-deploy
```

> **What this does:** When GitHub Actions connects with this key and passes `main` or `staging`
> as the SSH command, the server runs `deploy.sh main` or `deploy.sh staging` regardless of
> what command was actually sent. The key cannot be used for interactive login, port forwarding,
> or anything else. `${SSH_ORIGINAL_COMMAND#* }` strips the command name and passes only the
> branch argument through to the script.

## Step 5 - Add GitHub Repository Secrets

GitHub Actions stores sensitive values as encrypted secrets that are injected into the workflow
as environment variables at runtime. They are never visible in logs.

Go to your GitHub repository → **Settings** → **Secrets and variables** → **Actions** →
**New repository secret**.

Add the following secrets one by one:

| Secret name | Value |
|---|---|
| `SSH_PRIVATE_KEY` | The full contents of `~/.ssh/github_actions_deploy` (the private key, including the `-----BEGIN...` and `-----END...` lines) |
| `SSH_HOST` | `<10.0.0.1>` (your server's IPv4 address) |
| `SSH_USER` | `holu` |

> **How to copy the private key:** On your local machine, run:
> ```bash
> cat ~/.ssh/github_actions_deploy
> ```
> Select all output including the header and footer lines and paste it as the secret value.

## Step 6 - Create the GitHub Actions Workflows

You will create two workflow files: one for production (triggered by pushes to `main`) and one
for staging (triggered by pushes to `staging`). Both are nearly identical — only the branch
name and the environment label differ.

In your Laravel project, create the workflows directory:

```bash
mkdir -p .github/workflows
```

### Step 6.1 - Production Workflow (main branch)

Create `.github/workflows/deploy-production.yml`:

```yaml
name: Deploy to Production

# ── Trigger ──────────────────────────────────────────────────────────────────
# This workflow runs only when code is pushed to the main branch.
# Pull requests, pushes to other branches, and tag pushes do NOT trigger this.
on:
  push:
    branches:
      - main

# ── Concurrency ───────────────────────────────────────────────────────────────
# If two pushes land in quick succession, cancel the older in-progress
# deployment before starting the new one. This prevents two deployments
# from running simultaneously and corrupting the release directory.
concurrency:
  group: deploy-production
  cancel-in-progress: true

jobs:
  deploy:
    name: Deploy (production)
    runs-on: ubuntu-latest

    # ── Environment ─────────────────────────────────────────────────────────
    # Linking the job to a GitHub Environment named "production" enables
    # the deployment history tab on your repository and lets you add
    # required reviewers who must approve before the job runs.
    environment:
      name: production
      url: https://<example.com>

    steps:
      # 1. Check out the repository so the runner has the code.
      #    fetch-depth: 1 is a shallow clone — faster than a full history.
      - name: Checkout code
        uses: actions/checkout@v4
        with:
          fetch-depth: 1

      # 2. Add the server's host key to known_hosts so SSH does not prompt
      #    "Are you sure you want to continue connecting?" and fail.
      - name: Add server to known hosts
        run: |
          mkdir -p ~/.ssh
          ssh-keyscan -H ${{ secrets.SSH_HOST }} >> ~/.ssh/known_hosts

      # 3. Write the private key from GitHub Secrets to a temporary file
      #    and lock down its permissions. SSH refuses to use a key file
      #    that is readable by others.
      - name: Configure SSH key
        run: |
          echo "${{ secrets.SSH_PRIVATE_KEY }}" > ~/.ssh/deploy_key
          chmod 600 ~/.ssh/deploy_key

      # 4. SSH into the server and trigger the deployment script.
      #    The script receives "main" as its argument so it knows which
      #    branch to clone. The restricted authorized_keys entry on the
      #    server intercepts the argument and passes it to deploy.sh.
      - name: Run deployment script on server
        run: |
          ssh -i ~/.ssh/deploy_key \
              -o StrictHostKeyChecking=no \
              ${{ secrets.SSH_USER }}@${{ secrets.SSH_HOST }} \
              "main"

      # 5. Notify in the Actions log whether the deployment succeeded.
      #    The "if: failure()" condition runs this step only when a
      #    previous step failed — useful for adding Slack or email alerts.
      - name: Notify on failure
        if: failure()
        run: |
          echo "Deployment to production FAILED for commit ${{ github.sha }}"
          echo "Check the server logs: tail -50 /var/www/your-app/shared/storage/logs/laravel.log"
```

### Step 6.2 - Staging Workflow (staging branch)

Create `.github/workflows/deploy-staging.yml`:

```yaml
name: Deploy to Staging

on:
  push:
    branches:
      - staging   # Only pushes to the staging branch trigger this workflow

concurrency:
  group: deploy-staging
  cancel-in-progress: true

jobs:
  deploy:
    name: Deploy (staging)
    runs-on: ubuntu-latest

    environment:
      name: staging
      url: https://<staging.example.com>

    steps:
      - name: Checkout code
        uses: actions/checkout@v4
        with:
          fetch-depth: 1

      - name: Add server to known hosts
        run: |
          mkdir -p ~/.ssh
          ssh-keyscan -H ${{ secrets.SSH_HOST }} >> ~/.ssh/known_hosts

      - name: Configure SSH key
        run: |
          echo "${{ secrets.SSH_PRIVATE_KEY }}" > ~/.ssh/deploy_key
          chmod 600 ~/.ssh/deploy_key

      - name: Run deployment script on server
        run: |
          ssh -i ~/.ssh/deploy_key \
              -o StrictHostKeyChecking=no \
              ${{ secrets.SSH_USER }}@${{ secrets.SSH_HOST }} \
              "staging"   # Passes "staging" to deploy.sh as the branch argument
```

### Step 6.3 - Set Up GitHub Environments

Setting up named Environments in GitHub gives you a deployment history panel and the option
to require manual approval before production deploys.

Go to your repository → **Settings** → **Environments** → **New environment**.

Create two environments:

**Production:**
- Name: `production`
- Under **Deployment protection rules**, enable **Required reviewers** if you want someone to
  approve every production push before the workflow runs. Leave it off for fully automatic
  deployments.
- Under **Environment secrets**, you can override `SSH_HOST` here if production and staging
  are on different servers. For this tutorial they share one server, so the repository-level
  secret is sufficient.

**Staging:**
- Name: `staging`
- No required reviewers — staging should deploy automatically so developers can see their
  changes immediately.

### Step 6.4 - Set Up the Staging Directory on the Server

The staging deployment uses the same `deploy.sh` script but clones the `staging` branch into
a separate releases tree. The staging Nginx server block points to its own `current` symlink.

Create the staging directory structure on the server:

```bash
sudo mkdir -p /var/www/your-app-staging/releases
sudo mkdir -p /var/www/your-app-staging/shared/storage/app/public
sudo mkdir -p /var/www/your-app-staging/shared/storage/framework/{cache,sessions,views}
sudo mkdir -p /var/www/your-app-staging/shared/storage/logs
sudo chown -R holu:holu /var/www/your-app-staging
```

Create a separate `.env` for staging:

```bash
nano /var/www/your-app-staging/shared/.env
```

Set staging-specific values:

```dotenv
APP_ENV=staging
APP_DEBUG=true
APP_URL=https://<staging.example.com>
DB_DATABASE=your_app_staging
# ... rest of your env values pointing to the staging database
```

Update `deploy.sh` to handle the staging path. Open the script and change the `APP_DIR` line
to be dynamic based on the branch argument:

```bash
nano /var/www/your-app/scripts/deploy.sh
```

Replace the configuration block at the top of the script:

```bash
# ── Configuration ────────────────────────────────────────────────────────────
BRANCH="${1:-main}"

if [ "$BRANCH" = "staging" ]; then
  APP_DIR="/var/www/your-app-staging"
else
  APP_DIR="/var/www/your-app"
fi

REPO="git@github.com:your-username/your-laravel-app.git"
KEEP_RELEASES=5
TIMESTAMP=$(date +"%Y%m%d%H%M%S")
RELEASE_DIR="$APP_DIR/releases/$TIMESTAMP"
SHARED_DIR="$APP_DIR/shared"
CURRENT_LINK="$APP_DIR/current"
# ─────────────────────────────────────────────────────────────────────────────
```

Create the staging Nginx server block:

```bash
sudo nano /etc/nginx/sites-available/your-app-staging
```

```nginx
server {
    listen 80;
    listen [::]:80;

    server_name <staging.example.com>;
    root /var/www/your-app-staging/current/public;

    index index.php;
    charset utf-8;

    location / {
        try_files $uri $uri/ /index.php?$query_string;
    }

    location ~ /\.(?!well-known).* {
        deny all;
    }

    location ~ \.php$ {
        fastcgi_pass unix:/run/php/php8.3-fpm.sock;
        fastcgi_param SCRIPT_FILENAME $realpath_root$fastcgi_script_name;
        include fastcgi_params;
    }

    access_log /var/log/nginx/your-app-staging.access.log;
    error_log  /var/log/nginx/your-app-staging.error.log;
}
```

Enable it and obtain a staging SSL certificate:

```bash
sudo ln -s /etc/nginx/sites-available/your-app-staging /etc/nginx/sites-enabled/
sudo nginx -t && sudo systemctl reload nginx
sudo certbot --nginx -d <staging.example.com>
```

## Step 8 - Roll Back to the Previous Release

If a deployment introduces a bug, rolling back is one command. The previous release is still
on disk (because `KEEP_RELEASES=5` keeps five releases). You just re-point the symlink.

Find the available releases:

```bash
ls -1 /var/www/your-app/releases/
```

Output example:

```
20250509091500
20250516143000
```

Point `current` back to the previous release:

```bash
ln -sfn /var/www/your-app/releases/20250509091500 /var/www/your-app/current
```

Restart queue workers so they pick up the old code:

```bash
php /var/www/your-app/current/artisan queue:restart
```

> Nginx does not need to be reloaded. The `root` directive already points to
> `/var/www/your-app/current/public`, and `current` now points to the old release.
> The switch is instant.

If the old release has a different database schema (because the new deployment ran a migration),
you may need to reverse the migration:

```bash
php /var/www/your-app/current/artisan migrate:rollback --force
```

## Step 9 - Automated Encrypted MySQL Backups to Hetzner Storage Box

Your database is the most valuable thing on your server. The server itself is replaceable;
the data is not. You will now set up automated nightly backups that:

1. Dump the MySQL database with `mysqldump`
2. Compress the dump with `gzip`
3. Encrypt the compressed dump with GPG using a symmetric passphrase
4. Upload the encrypted file to your Hetzner Storage Box over SSH
5. Prune backups older than 7 days from the Storage Box automatically

Even if your server is destroyed, your last 7 days of data are safe on a separate Hetzner
system.

### Step 9.1 - Enable SSH Access on Your Storage Box

Log into the [Hetzner Robot panel](https://robot.hetzner.com) → **Storage Box** → select
your box → **Settings**.

Enable **SSH support** and **External reachability** (needed to connect from your server).

Generate a dedicated SSH key for the backup process on your server:

```bash
ssh-keygen -t ed25519 -C "storagebox-backup" -f ~/.ssh/storagebox_backup
```

No passphrase — this key will be used by a non-interactive cron job.

Add the public key to your Storage Box. Hetzner Storage Boxes use port 23:

```bash
ssh-copy-id -p 23 -i ~/.ssh/storagebox_backup.pub u123456@u123456.your-storagebox.de
```

You will be prompted for your Storage Box password (not your Hetzner account password — check
the Robot panel for the Storage Box password).

Test the connection:

```bash
ssh -p 23 -i ~/.ssh/storagebox_backup u123456@u123456.your-storagebox.de
```

You should get a Storage Box shell prompt. Type `exit` to leave.

### Step 9.2 - Create a MySQL User for Backups

On the server, log into MySQL and create a read-only user for the backup process. This user
can dump data but cannot modify it:

```bash
sudo mysql
```

```sql
CREATE USER 'backup_user'@'localhost' IDENTIFIED BY 'BackupPassword1!';
GRANT SELECT, LOCK TABLES, SHOW VIEW, EVENT, TRIGGER ON laravel_db.* TO 'backup_user'@'localhost';
FLUSH PRIVILEGES;
EXIT;
```

Store the credentials in a `.my.cnf` file so the backup script does not have a password in
plaintext in the script itself or in the process list:

```bash
nano ~/.my.cnf
```

```ini
[mysqldump]
user=backup_user
password=BackupPassword1!
```

Lock down the file:

```bash
chmod 600 ~/.my.cnf
```

### Step 9.3 - Create the Backup Script

```bash
nano /var/www/your-app/scripts/backup-db.sh
```

```bash
#!/usr/bin/env bash
set -euo pipefail

# ── Configuration ────────────────────────────────────────────────────────────
DB_NAME="laravel_db"
BACKUP_DIR="/tmp/db-backups"
DATE=$(date +"%Y-%m-%d_%H-%M-%S")
FILENAME="${DB_NAME}_${DATE}.sql.gz.gpg"
STORAGE_BOX_USER="u123456"
STORAGE_BOX_HOST="u123456.your-storagebox.de"
STORAGE_BOX_PATH="/backups/mysql"   # Directory inside your Storage Box
KEEP_DAYS=7
GPG_PASSPHRASE_FILE="/home/holu/.backup-passphrase"  # See Step 9.4
# ─────────────────────────────────────────────────────────────────────────────

# Create a temp working directory.
mkdir -p "$BACKUP_DIR"

echo "==> Dumping database '$DB_NAME'"

# 1. Dump, compress, and encrypt in a single pipeline.
#
#    mysqldump: credentials come from ~/.my.cnf (no password in command).
#    gzip:      compresses the SQL file. A typical Laravel DB compresses to
#               5-10% of its raw size.
#    gpg:       symmetric encryption with the passphrase from a file.
#               --batch and --yes suppress interactive prompts.
#               The output file has the .gpg extension.
mysqldump \
  --single-transaction \
  --routines \
  --triggers \
  "$DB_NAME" \
  | gzip \
  | gpg --symmetric \
        --cipher-algo AES256 \
        --batch \
        --yes \
        --passphrase-file "$GPG_PASSPHRASE_FILE" \
        --output "$BACKUP_DIR/$FILENAME"

echo "==> Uploading to Storage Box"

# 2. Create the target directory on the Storage Box (silently, in case it
#    already exists), then upload the encrypted backup file.
ssh -p 23 \
    -i /home/holu/.ssh/storagebox_backup \
    -o StrictHostKeyChecking=no \
    "${STORAGE_BOX_USER}@${STORAGE_BOX_HOST}" \
    "mkdir -p ${STORAGE_BOX_PATH}"

rsync -az \
  -e "ssh -p 23 -i /home/holu/.ssh/storagebox_backup" \
  "$BACKUP_DIR/$FILENAME" \
  "${STORAGE_BOX_USER}@${STORAGE_BOX_HOST}:${STORAGE_BOX_PATH}/"

echo "==> Pruning backups older than $KEEP_DAYS days on Storage Box"

# 3. List .gpg files on the Storage Box sorted by modification time, skip
#    the newest KEEP_DAYS worth, and delete the rest. This runs entirely
#    on the Storage Box via SSH so no file transfer is needed.
ssh -p 23 \
    -i /home/holu/.ssh/storagebox_backup \
    "${STORAGE_BOX_USER}@${STORAGE_BOX_HOST}" \
    "find ${STORAGE_BOX_PATH} -name '*.gpg' -mtime +${KEEP_DAYS} -delete"

echo "==> Cleaning up local temp file"
rm -f "$BACKUP_DIR/$FILENAME"

echo "==> Backup complete: $FILENAME"
```

Make it executable:

```bash
chmod +x /var/www/your-app/scripts/backup-db.sh
```

### Step 9.4 - Store the GPG Passphrase Securely

The encryption passphrase must not be in the script. Store it in a file readable only by
`holu`:

```bash
# Generate a strong random 32-character passphrase
openssl rand -base64 32 > /home/holu/.backup-passphrase
chmod 600 /home/holu/.backup-passphrase
```

> **Write this passphrase down and store it somewhere safe offline** — a password manager,
> a printed sheet in a physical safe, or a secrets vault. Without it, your backups cannot be
> decrypted. The passphrase is not stored anywhere on the Storage Box.

### Step 9.5 - Schedule the Backup with Cron

Open the cron editor for `holu`:

```bash
crontab -e
```

Add this line to run the backup every night at 02:30 server time and log the output:

```
30 2 * * * /var/www/your-app/scripts/backup-db.sh >> /var/www/your-app/shared/storage/logs/backup.log 2>&1
```

> **Why 02:30?** Most production traffic is lowest in the early hours. `mysqldump` with
> `--single-transaction` does not lock tables for InnoDB (which Laravel uses by default), so
> this is not strictly necessary — but it avoids adding any read load during peak hours.

### Step 9.6 - Test the Backup and Restore Process

Run the backup script manually to confirm it works:

```bash
/var/www/your-app/scripts/backup-db.sh
```

Verify the file appeared on your Storage Box:

```bash
ssh -p 23 -i ~/.ssh/storagebox_backup u123456@u123456.your-storagebox.de \
  "ls -lh /backups/mysql/"
```

**Test that you can actually restore from the backup.** A backup you have never tested is not
a backup. Copy the file back to the server and decrypt it:

```bash
# Download the backup file from the Storage Box
scp -P 23 -i ~/.ssh/storagebox_backup \
  u123456@u123456.your-storagebox.de:/backups/mysql/laravel_db_2025-05-16_02-30-00.sql.gz.gpg \
  /tmp/

# Decrypt it
gpg --batch \
    --passphrase-file /home/holu/.backup-passphrase \
    --output /tmp/laravel_db_restore.sql.gz \
    --decrypt /tmp/laravel_db_2025-05-16_02-30-00.sql.gz.gpg

# Decompress it
gunzip /tmp/laravel_db_restore.sql.gz

# Inspect the first few lines to confirm it is valid SQL
head -20 /tmp/laravel_db_restore.sql
```

You should see SQL output that begins with `-- MySQL dump ...`. The restore command (to a
test database, not your live one) would be:

```bash
sudo mysql -e "CREATE DATABASE laravel_db_test;"
sudo mysql laravel_db_test < /tmp/laravel_db_restore.sql
```

## Step 10 - Verify the Complete Pipeline End to End

With everything in place, here is what happens when you push code:

1. You push a commit to `main` on GitHub.
2. GitHub Actions triggers `deploy-production.yml`.
3. The runner SSHes into `<10.0.0.1>` using the deploy key and passes `"main"` as the command.
4. The restricted `authorized_keys` entry runs `deploy.sh main` on the server.
5. `deploy.sh` clones the `main` branch into `/var/www/your-app/releases/TIMESTAMP/`.
6. Shared `.env` and `storage/` are symlinked in. Composer installs. Migrations run.
7. Config, routes, and views are cached.
8. `ln -sfn` makes the new release live in one atomic operation.
9. `queue:restart` signals workers to reload gracefully.
10. Old releases beyond the last 5 are pruned.
11. At 02:30 the following night, the backup script dumps, encrypts, and ships the database to
    the Storage Box.

Check the Actions tab on GitHub to see the deployment history. Each deployment is linked to
the commit that triggered it, so you always know exactly what code is live.

## Conclusion

You now have a deployment pipeline that has no single-vendor dependency, no monthly fee, and
no downtime. The atomic symlink pattern has been the foundation of every serious deployment
tool — Capistrano, Deployer, Envoyer — for over a decade. This tutorial gives you the same
result with plain bash and GitHub Actions.

The key points worth remembering:

- **Migrations run before the symlink flips.** Write migrations that are backward-compatible
  with the current release (add columns, do not rename or drop them in the same deploy). This
  is what makes the zero-downtime claim true for database changes as well as code changes.
- **The backup is only as good as a test restore.** Run `Step 9.6` again in three months to
  confirm your decrypt passphrase is still accessible and the backup format is still valid.
- **The deploy key is restricted.** Even if the private key leaks, it cannot be used for
  anything other than triggering the deployment script with a branch name argument.

**Next steps:**

- Add a Slack or email notification step to the workflow's `on: failure` condition so the team
  is alerted immediately when a deployment fails.
- Extend the backup script to also back up the `shared/storage/app/` directory (user-uploaded
  files) using the same rsync-to-Storage-Box pattern.
- Add a `php artisan down` / `php artisan up` wrapper around the migration step if you have
  migrations that are not backward-compatible and need a short maintenance window.
